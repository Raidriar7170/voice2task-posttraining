Must Fix Phase 2: strict contract status

Status: completed on branch, verification passed

结论：Phase 2 现在完成了 strict parser/status 的收口：主评测先判断单一根 JSON 解析，再判断 strict schema，再判断 semantic contract，最后才允许 exact match；unknown nested fields 也会在中心化校验里直接失败。

本阶段改动

  • as_contract / BrowserTaskContract.from_dict / validate_contract_status 统一走 strict unknown-field rejection，不再静默丢弃额外字段。
  • safety 现在必须且只能包含 {"allow","reason"}；safety.debug 之类的 nested unknown key 会让 strict schema 直接失败。
  • evaluate_predictions 明确拆成 json_parse_rate、strict_schema_valid_rate、semantic_contract_valid_rate、contract_exact_match；仅保留 json_valid_rate 作为兼容别名。
  • contract_exact_match 只在 strict valid + semantic valid + canonical prediction==gold 时递增；语义矛盾但字符串相同的 contract 不再算 exact。
  • zero-support safety 指标改成 null：预测正例支持为 0 时 safety_precision=null，gold 正例支持为 0 时 safety_recall=null，并额外报告 safety_tp/fp/fn 与 support；report 聚合层不再把它们压回 0.0。
  • layered residual diagnosis 保留显式命名的 permissive diagnostic parser，但 primary scoring 仍只认 strict parser。
  • 下游报告表格补齐 parse / strict schema / semantic / exact 的指标梯度，不再只展示旧的 json_valid_rate 兼容列。
  • 最后 hardening 补上 malformed primitive 边界：task_type 和 route 必须先通过 string 类型检查，[] / {} 不再触发 TypeError。
